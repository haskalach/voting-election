# Evaluation Framework

**How to measure AI-assisted development success.**

---

## Framework Overview

### The 4 Dimensions of Evaluation

We measure AI-assisted development across 4 dimensions:

1. **Adoption** — Are people using AI tools?
2. **Productivity** — Is work getting done faster?
3. **Quality** — Is code better (secure, tested, documented)?
4. **Experience** — Are people happy using these tools?

### The Measurement Process

```
Collect → Analyze → Plan → Implement → Measure
  ↑                                        ↓
  └────────────────────────────────────────┘
```

---

## 1. Adoption Metrics

**Question**: Are developers using AI tools?

### What to Measure

| Metric                  | How                    | Frequency | Target         |
| ----------------------- | ---------------------- | --------- | -------------- |
| **Active Users**        | % team using tools     | Monthly   | >80%           |
| **Usage Frequency**     | Times per week         | Monthly   | >5 times       |
| **Tool Preference**     | Which tools most used  | Monthly   | Know top 3     |
| **Feature Adoption**    | Which features used    | Monthly   | >60% trying    |
| **Training Completion** | % completed onboarding | One-time  | 100%           |
| **Integration Depth**   | Part of daily workflow | Quarterly | >90% daily use |

### How to Collect

**GitHub**: Commit message patterns, PR activity timing

```
- Look for patterns: "@copilot", "/prompt", "generated by AI"
- Tool timing: Do PRs include AI-generated code?
- Feedback: Do PRs reference instruction files?
```

**Surveys**: Monthly question

```
"How many times did you use AI tools this month?"
- 0 times
- 1-5 times
- 6-10 times
- 10+ times (Daily user)
```

**Tool Analytics** (if available):

```
- Copilot usage in VS Code
- Prompt template downloads
- GitHub Actions runs
```

**Direct Observation**:

```
- Code reviews: Do PRs follow AI-suggested patterns?
- Tests: Are new test suites using templates?
- Docs: Are new docs using documentation prompt?
```

### Red Flags

🚩 Low adoption (<50%)

- Why? (Survey to find out)
- Missing training? → Run workshop
- Tools not helpful? → Gather feedback, improve
- Friction in workflow? → Simplify process

## 2. Productivity Metrics

**Question**: Is work getting done faster?

### What to Measure

| Task                  | Baseline | Target     | Metric                    |
| --------------------- | -------- | ---------- | ------------------------- |
| **Code Review**       | 30 min   | 5 min      | Time saved per review     |
| **Test Generation**   | 60 min   | 10 min     | Time saved per test suite |
| **Documentation**     | 120 min  | 15 min     | Time saved per doc        |
| **API Design**        | 90 min   | 20 min     | Time saved per API        |
| **Refactoring**       | 120 min  | 30 min     | Time saved per refactor   |
| **Velocity (Sprint)** | Current  | +20%       | Features per sprint       |
| **Code Review Cycle** | Current  | 30% faster | Hours in review state     |

### How to Collect

**Time Tracking** (Self-reported):

```
"How long did code review take?"
- With AI tools: 5 minutes
- Without AI tools: 30 minutes
- Savings: 25 minutes per review
```

**Velocity Metrics** (Automatic):

```
- Commits per week (before vs after AI)
- PRs merged per week
- Lines of code (from commits)
- Tests written per sprint
```

**GitHub Metrics** (Automatic):

```
- Average time in review (PR to merge)
- Commit frequency (commits per day per developer)
- PR size (lines changed per PR)
```

**Manual Tracking** (Spreadsheet):

```
Week 1: Code review with AI = 5 min, without = 30 min
Week 2: Code review with AI = 4 min, without = 28 min
Week 3: Code review with AI = 6 min, without = 32 min
Average: 5 min vs 30 min = 6x faster
```

### Calculation Example

**Code Review Productivity**:

```
Baseline: 30 min per review
Current: 5 min per review
Savings per review: 25 min

Reviews per week: 10
Weekly savings: 10 × 25 = 250 min (4 hours)
Monthly savings: 4 × 4 = 16 hours/month per developer

Team (5 developers): 5 × 16 = 80 hours/month
Annual savings: 80 × 12 = 960 hours

Cost savings: 960 hours × $150/hr = $144,000/year
```

## 3. Quality Metrics

**Question**: Is code/tests/docs better?

### What to Measure

| Dimension            | Metric                   | Baseline | Target        |
| -------------------- | ------------------------ | -------- | ------------- |
| **Test Coverage**    | % lines covered          | 100%     | Maintain 100% |
| **Defect Rate**      | Bugs found after release | Current  | -25%          |
| **Security Issues**  | Critical vulns           | 0        | 0             |
| **Code Quality**     | Build errors             | 0        | 0             |
| **Documentation**    | API docs complete        | Baseline | 100%          |
| **Code Review Gaps** | Issues missed in review  | Baseline | -40%          |
| **Technical Debt**   | Issues in backlog        | Count    | -30%          |

### How to Collect

**Code Coverage** (Automatic):

```
- dotnet test with coverage reports
- Verify: 100% coverage maintained
- Track monthly
```

**Security Issues** (Automatic):

```
- GitHub security scanning
- OWASP Top 10 checks
- SonarQube or similar
- Target: 0 critical issues
```

**Defect Rate** (From bug reports):

```
- Count bugs found after release
- Compare to baseline (before AI)
- Target: 25% reduction

Example:
Pre-AI:  10 bugs/month found in production
Post-AI: 7 bugs/month found in production
Improvement: 30% fewer escapes
```

**Documentation Completeness** (Manual):

```
- Count methods with complete docs
- Target: 100% of public APIs
- Tracked monthly
```

## 4. Experience Metrics

**Question**: Are people happy with AI tools?

### What to Measure

| Metric              | How                      | Frequency | Target              |
| ------------------- | ------------------------ | --------- | ------------------- |
| **Satisfaction**    | NPS (Net Promoter Score) | Monthly   | >50                 |
| **Ease of Use**     | 1-5 rating               | Monthly   | >4.0/5              |
| **Value Delivered** | 1-5 rating               | Monthly   | >4.2/5              |
| **Learning Curve**  | Hours to proficiency     | One-time  | <4 hours            |
| **Pain Points**     | Top 3 issues             | Monthly   | Trending down       |
| **Suggestions**     | Improvement ideas        | Monthly   | Track and implement |

### How to Collect

**Monthly Survey** (5 questions, 3 minutes):

```
1. How many times did you use AI tools this month? (Adoption)
2. How satisfied are you with AI tools? 1-10 (NPS)
3. How easy are they to use? 1-5 (Ease)
4. How much value do they deliver? 1-5 (Value)
5. Top suggestion for improvement? (Open-ended)
```

**NPS Calculation**:

```
Promoters (9-10): Likely to recommend
Passives (7-8): Neutral
Detractors (1-6): Unlikely to recommend

NPS = (Promoters - Detractors) / Total × 100

Example:
100 developers surveyed
70 Promoters (70%)
20 Passives (20%)
10 Detractors (10%)

NPS = (70 - 10) / 100 × 100 = 60 (Good!)
```

**Feedback Collection**:

```
"What's your top suggestion for improvement?"
- Better prompt templates (3 votes)
- Easier Copilot integration (2 votes)
- More examples (1 vote)
- Security scanning in Copilot (1 vote)

Action: Create better templates (highest priority)
```

## Analysis & Insights

### Question to Ask Monthly

1. **Adoption**: Rising or flat? Why?
   - If rising: What features drive adoption?
   - If flat: What's the barrier?

2. **Productivity**: Faster or same?
   - If faster: Which tasks improved?
   - If flat: Are people using tools correctly?

3. **Quality**: Better or same?
   - If better: Which improvements helped?
   - If flat: What quality issues remain?

4. **Experience**: Happy or frustrated?
   - If happy: What works best?
   - If frustrated: What to fix first?

### Data-Driven Decisions

**Example Decision Making**:

```
Data: Adoption is 80%, but 3 developers not using tools
Action: Interview those 3 developers
Finding: "Don't understand how to use prompts"
Solution: Create better templates, add examples
Outcome: Next month, adoption rises to 85%
```

## Red Flags & Escalation

### When to Escalate

🚩 **Adoption < 50%**

- Team doesn't understand tools
- Tools don't fit workflow
- Training needed
  → Action: Workshop, hands-on training

🚩 **NPS < 30**

- Tools are frustrating
- Value not clear
- Major pain points exist
  → Action: Deep investigation, major fixes

🚩 **Quality dropping**

- AI-generated code has issues
- Tests are missing edge cases
- Documentation incomplete
  → Action: Review templates, add guardrails, retrain

🚩 **Productivity flat**

- Tools not being used enough
- Learning curve too high
- Wrong tool for job
  → Action: Better integration, simpler workflows
