## Architecture Principles

### Layered Architecture

```
API Layer (Controllers)
    ↓
Application Layer (DTOs, Services)
    ↓
Domain Layer (Entities, Interfaces)
    ↓
Infrastructure Layer (Repositories, DbContext)
    ↓
Database
```

### Key Patterns

- **Repository Pattern**: All data access via repositories
- **Dependency Injection**: Constructor injection throughout
- **Unit of Work**: Coordinated transactions
- **Async/Await**: Never sync calls in services/controllers
- **Record Types**: DTOs use C# records (immutable)

### Naming Conventions

**Classes**

```csharp
// Services: verb + Service
public class AuthService : IAuthService { }
public class OrganizationService : IOrganizationService { }

// Repositories: Noun + Repository
public class UserRepository : IUserRepository { }
public class OrganizationRepository : IOrganizationRepository { }

// Controllers: Plural noun + Controller
public class OrganizationsController : ControllerBase { }
public class EmployeesController : ControllerBase { }
```

**Methods**

```csharp
// Async methods end with Async
Task<UserDto> GetUserAsync(int id);
Task<bool> CreateAsync(CreateUserDto dto);

// Query methods: Get or Find
Task<User> GetByIdAsync(int id);
Task<User> GetByEmailAsync(string email);

// Mutation methods: Create, Update, Delete
Task<T> CreateAsync(TCreateDto dto);
Task<T> UpdateAsync(int id, TUpdateDto dto);
Task DeleteAsync(int id);
```

**DTOs**

```csharp
// Request DTOs follow action naming
public record CreateOrganizationDto(...);
public record UpdateOrganizationDto(...);
public record LoginRequestDto(...);

// Response DTOs are entity-named
public record OrganizationDto(...);
public record UserDto(...);
public record LoginResponseDto(...);
```

### Code Style Rules

**1. Async/Await**

```csharp
// ✅ GOOD: Async throughout
public async Task<UserDto> GetUserAsync(int id)
{
    return await _userRepository.GetByIdAsync(id);
}

// ❌ BAD: Sync over async
public UserDto GetUser(int id)
{
    return _userRepository.GetByIdAsync(id).Result;
}
```

**2. Dependency Injection**

```csharp
// ✅ GOOD: Constructor injection
public class OrganizationService
{
    private readonly IOrganizationRepository _orgRepo;

    public OrganizationService(IOrganizationRepository orgRepo)
        => _orgRepo = orgRepo;
}

// ❌ BAD: Service locator pattern
public class OrganizationService
{
    public void DoSomething()
    {
        var repo = ServiceLocator.GetService<IOrganizationRepository>();
    }
}
```

**3. Immutable DTOs**

```csharp
// ✅ GOOD: Records are immutable
public record CreateUserDto(
    [Required][EmailAddress] string Email,
    [Required][MinLength(8)] string Password,
    [Required] string FirstName,
    [Required] string LastName
);

// ❌ BAD: Mutable classes
public class CreateUserDto
{
    public string Email { get; set; }  // Mutable
    public string Password { get; set; }
}
```

**4. Validation**

```csharp
// ✅ GOOD: DataAnnotations on DTOs
public record LoginRequestDto(
    [Required][EmailAddress] string Email,
    [Required][MinLength(8)] string Password
);

// API validates automatically via ModelState check
if (!ModelState.IsValid)
    return BadRequest(ModelState);

// ❌ BAD: Manual validation in service
public async Task<LoginResponseDto> LoginAsync(LoginRequestDto request)
{
    if (string.IsNullOrEmpty(request.Email))  // Don't do this
        throw new ArgumentException("Email required");
}
```

**5. Exception Handling**

```csharp
// Services throw domain exceptions
throw new KeyNotFoundException("User not found");
throw new InvalidOperationException("Duplicate email");
throw new UnauthorizedAccessException("Invalid password");

// Controllers catch and translate to HTTP responses
catch (KeyNotFoundException) => NotFound()
catch (InvalidOperationException ex) => BadRequest(ex.Message)
catch (UnauthorizedAccessException) => Unauthorized()
```

**6. XML Documentation**

```csharp
// ✅ GOOD: Comprehensive documentation
/// <summary>Authenticates user with email and password</summary>
/// <param name="request">Login credentials</param>
/// <returns>JWT access token and refresh token</returns>
/// <exception cref="InvalidOperationException">Invalid credentials</exception>
/// <response code="200">Authentication successful</response>
/// <response code="401">Invalid credentials</response>
public async Task<LoginResponseDto> LoginAsync(LoginRequestDto request) { }

// ❌ BAD: Missing documentation
public async Task<LoginResponseDto> LoginAsync(LoginRequestDto request) { }
```

## Security Best Practices

### Authentication

- ✅ JWT tokens with expiration
- ✅ Refresh tokens stored in HTTP-only cookies (frontend: storage only)
- ✅ Access tokens in-memory only (Angular signal)
- ✅ Rate limiting on auth endpoints (10 req/min)
- ❌ Never store access tokens in localStorage
- ❌ Never hardcode secrets

### Authorization

- ✅ [Authorize(Roles = "...")] attributes on all protected endpoints
- ✅ Organization isolation checks for Managers
- ✅ User ID validation from claims
- ✅ Audit logging on all mutations
- ❌ Never trust user input for authorization

### Data Protection

- ✅ Passwords hashed with BCrypt
- ✅ Secrets in external configuration (not in code)
- ✅ CORS configured for frontend origin only
- ✅ HTTPS enforced in production
- ❌ Never log passwords, tokens, or PII

## What NOT to Do

### ❌ Don't

- Add external dependencies without approval
- Use `Task.Wait()` or `.Result` (causes deadlocks)
- Hardcode configuration values
- Mix sync and async code
- Create tight coupling between layers
- Skip validation or error handling
- Modify database schema without migrations
- Use magic strings for role names (use constants)

### ✅ Do

- Use async/await throughout
- Inject dependencies via constructor
- Validate input at API layer
- Use records for immutable data
- Write comprehensive tests
- Document public methods with XML
- Use migrations for schema changes
- Centralize constants and configurations
